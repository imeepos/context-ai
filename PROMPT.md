# Harness Engineering 核心指南

> 来源：OpenAI、Anthropic、Stripe、Martin Fowler 等团队实践总结

---

## 一、核心定义

**Harness Engineering** 是围绕 AI Agent 设计和构建约束机制、反馈回路、工作流控制和持续改进循环的系统工程实践。

**核心理念**："每当发现 Agent 犯错，就花时间设计一个解决方案，确保 Agent 永远不会再犯同样的错误。" — Mitchell Hashimoto

---

## 二、为什么需要 Harness Engineering

### 2.1 瓶颈在基础设施，不在模型智能

- **Can.ac 实验**：仅改变 Harness 的工具格式，Grok Code Fast 1 从 6.7% 跃升至 68.3%
- **LangChain 实验**：同一模型通过 Harness 改进，排名从第 30 名升至第 5 名
- **结论**：在纠结模型选择之前，先审视 Harness 设计能获得更高的投资回报率

### 2.2 Agent 的四大典型失败模式

| 失败模式                | 描述                                  | 解决方案                   |
| ----------------------- | ------------------------------------- | -------------------------- |
| 一步到位 (One-shotting) | 试图一次做完所有事情，上下文耗尽      | 分层上下文 + 增量执行      |
| 过早宣布胜利            | 看到部分进展就宣布完成                | 结构化功能列表 + JSON 追踪 |
| 过早标记功能完成        | 写完代码就标记完成，无端到端测试      | 强制验证 + 测试步骤        |
| 环境启动困难            | 每次会话花费大量 token 弄清楚如何运行 | init.sh 脚本 + 进度文件    |

### 2.3 上下文窗口的甜蜜区间

- **Smart Zone（前约 40%）**：聚焦、准确的推理
- **Dumb Zone（超过约 40%）**：幻觉、循环、格式错误、低质量代码

**原则**：给 Agent 塞更多 token 不会让它更聪明，反而会让它变笨。

## 三、四大支柱

### 支柱一：上下文架构 (Context Architecture)

**核心原则**：Agent 应当恰好获得当前任务所需的上下文——不多不少。

| 层级                 | 加载时机                  | 内容示例              | 上下文占用 |
| -------------------- | ------------------------- | --------------------- | ---------- |
| Tier 1：会话常驻     | 每次会话自动加载          | AGENTS.md / CLAUDE.md | 最小       |
| Tier 2：按需加载     | 特定子 Agent 或技能调用时 | 专业化 Agent 上下文   | 中等       |
| Tier 3：持久化知识库 | Agent 主动查询时          | 研究文档、规格说明    | 按需       |

### 支柱二：Agent 专业化 (Agent Specialization)

**核心原则**：专注于特定领域、拥有受限工具的 Agent 优于拥有全部权限的通用 Agent。

| Agent 角色 | 职责范围               | 工具权限                |
| ---------- | ---------------------- | ----------------------- |
| 研究 Agent | 探索代码库、分析实现   | 只读 (Read, Grep, Glob) |
| 规划 Agent | 将需求分解为结构化任务 | 只读，无写入权限        |
| 执行 Agent | 实现单个具体任务       | 限定范围的读写权限      |
| 审查 Agent | 审计完成的工作         | 只读 + 标记权限         |
| 调试 Agent | 修复审查发现的问题     | 限定范围的修复权限      |
| 清理 Agent | 对抗熵积累             | 读写权限                |

### 支柱三：持久化记忆 (Persistent Memory)

**核心原则**：进度持久化在文件系统上，而非上下文窗口中。

**编码 Agent 的典型会话启动流程**：
1. 运行 `pwd` 查看工作目录
2. 读取 git log 和进度文件，了解最近的工作
3. 读取 feature list 文件，选择最高优先级的未完成功能
4. 启动开发服务器，运行基础端到端测试
5. 确认基本功能正常后，开始新功能开发

**关键发现**：使用 JSON 格式追踪 feature 状态比 Markdown 更有效。

### 支柱四：结构化执行 (Structured Execution)

**核心原则**：将思考与执行分离。

**执行序列**：理解 → 规划 → 执行 → 验证

**人工检查点的价值**：审查计划远比审查代码快。当规格正确时，实现自然可靠。

## 四、五大核心原则 (OpenAI 实践)

### 原则 1：设计环境，而非编写代码
当 Agent 卡住时，诊断"缺少什么能力"并让 Agent 自己构建该能力。

### 原则 2：机械化地执行架构约束
定义依赖方向并用自定义 Linter 和结构测试自动检测违规。
```
Types → Config → Repo → Service → Runtime → UI
```
**关键**：如果不能机械化地强制执行，Agent 就会偏离。

### 原则 3：将代码仓库作为唯一事实源
所有团队知识作为版本控制的制品放置在仓库中，不放在 Slack、Wiki 或 Google Docs。

### 原则 4：将可观测性连接到 Agent
- Chrome DevTools 集成
- DOM 快照和截图
- 日志和指标查询

### 原则 5：对抗熵
定期运行的"垃圾回收" Agent：
- 扫描文档不一致
- 检测架构约束违规
- 清理冗余或低质量代码

## 五、核心组件详解

### 5.1 AGENTS.md — Agent 的活文档

**关键特性**：
- 不是一次性编写后遗忘的静态文档
- 每当 Agent 犯错时都要更新——文档变成了反馈循环
- 简单的错误通过更新 AGENTS.md 解决
- 复杂的问题需要构建工具层面的解决方案

**OpenAI 进阶实践**：小型 AGENTS.md 指向更深层的事实源——设计文档、架构图、执行计划、质量评级——全部版本控制。

### 5.2 自定义 Linter 设计

**巧妙设计**：当 Agent 违反架构约束时，错误消息不仅标记违规——还告诉 Agent 如何修复。

```
❌ 传统 Linter: "Architecture violation: UI importing from Types"
✅ Harness Linter: "Architecture violation: UI cannot import from Types.
   Fix: Move shared types to a common module or use dependency injection"
```

### 5.3 初始化 Agent vs 编码 Agent

| 问题             | 初始化 Agent 行为       | 编码 Agent 行为                |
| ---------------- | ----------------------- | ------------------------------ |
| 过早宣布完成     | 建立功能列表 JSON 文件  | 读取功能列表，选择单个功能     |
| 未文档化的进度   | 编写 git 仓库和进度文件 | 读取进度文件和 git 日志        |
| 过早标记功能完成 | 建立功能列表文件        | 仅在仔细测试后才标记 "passing" |
| 环境启动困难     | 编写 init.sh 脚本       | 会话开始时读取 init.sh         |

## 六、最佳实践清单

### 立即可行的行动

- [ ] **创建并维护 AGENTS.md**：每当 Agent 犯错时都更新
- [ ] **在仓库中建立单一事实源**：所有知识作为版本控制制品
- [ ] **构建自定义 Linter**：在错误消息中嵌入修复指令
- [ ] **提供端到端测试工具**：浏览器自动化（如 Puppeteer MCP）
- [ ] **实施增量执行策略**：每次会话只处理一个功能
- [ ] **分层管理上下文**：使用 Tier 1/2/3 渐进式披露
- [ ] **上下文利用率保持在 40% 以下**
- [ ] **建立定期"垃圾回收"机制**

### Harness 成熟度评估

| 阶段    | 特征                          | 工程师角色               |
| ------- | ----------------------------- | ------------------------ |
| Level 0 | 无结构化约束                  | 手动写代码 + 偶尔使用 AI |
| Level 1 | AGENTS.md + 基础 Linter       | 主要写代码，AI 辅助      |
| Level 2 | CI/CD + 自动化测试 + 进度追踪 | 规划 + 审查为主          |
| Level 3 | 多 Agent 分工 + 分层上下文    | 环境设计 + 管理为主      |
| Level 4 | 无人值守并行化 + 自修复       | 架构师 + 质量把关者      |

## 七、六大业界共识

1. **瓶颈在基础设施，不在模型智能** — 6+ 来源支持
2. **文档必须是活的反馈循环，不是静态制品** — 每次失败都更新
3. **思考与执行必须分离** — 先规划再执行
4. **上下文不是越多越好** — 40% 是甜蜜区间
5. **约束必须机械化执行** — 不能靠文档记录
6. **工程师角色从"写代码"转向"设计环境 + 管理工作"**

> "永远不要让 Agent 在你审查和批准书面计划之前写代码。" — Boris Tane (Cloudflare)
